# 🔐 مراجعة أمنية — SpeedPro Dashboard

> تاريخ المراجعة: 2026-06-21 — مراجعة دفاعية لكود الداشبورد الخاص بك.

## ⚠️ القاعدة الأساسية
الداشبورد ده **تطبيق واجهة أمامية بالكامل** (HTML/JS) بيتصل بـ Firebase مباشرةً.
أي كود أو مفتاح أو باسورد في الملفات دي **مكشوف لأي زائر** عبر `View Source` / `F12`.
**مستحيل تخفي سر في كود الواجهة** — الأمان الحقيقي لازم يكون في **قواعد Firebase** أو في **سيرفر/خدمة وسيطة**.

---

## 🔴 ثغرات حرجة (لازم تتقفل فوراً)

### 1) قاعدة بيانات Firebase بدون مصادقة (أخطر ثغرة)
- التطبيق **مش بيستخدم Firebase Authentication**. بيقرأ `ninja_data/admins` (الأسماء + الباسوردات نص صريح) ويقارن في المتصفح.
- رابط القاعدة مكشوف: `ninja-system-30301-default-rtdb.firebaseio.com`
- **لو قواعد الأمان مفتوحة**، أي حد يقدر يحمّل كل القاعدة (كل الباسوردات + كل البيانات) بطلب واحد:
  `https://ninja-system-30301-default-rtdb.firebaseio.com/.json`
- وأخطر: لو الكتابة مفتوحة، أي حد يقدر **يعدّل أو يمسح كل بياناتك** أو يحقن بيانات خبيثة.

**الحل الحقيقي (موصى به):** الانتقال لـ Firebase Authentication (إيميل/باسورد للمشرفين)، وإلغاء تخزين الباسوردات في القاعدة، ثم قواعد أمان حسب الدور. ده تعديل معماري — أقدر أعمله لك كخطوة منفصلة.

**كحد أدنى فوري:** افتح Firebase Console → Realtime Database → Rules وحط قواعد تمنع القراءة/الكتابة المفتوحة. (ملاحظة: من غير Auth، أي قاعدة بتسمح للتطبيق بتسمح للمهاجم برضه — عشان كده الانتقال لـ Auth ضروري).

مثال قواعد بعد تفعيل Firebase Auth:
```json
{
  "rules": {
    ".read": "auth != null",
    ".write": "auth != null",
    "ninja_data": {
      "admins": {
        ".read":  "auth != null && root.child('ninja_data/admins/'+auth.uid+'/role').val() == 'super_admin'",
        ".write": "auth != null && root.child('ninja_data/admins/'+auth.uid+'/role').val() == 'super_admin'"
      }
    }
  }
}
```

### 2) ثغرة XSS مخزّنة (حقن سكريبت) — ✅ تم تقليلها بشكل كبير
- مثال الخطر: لو اتسجّل اسم مندوب = `<img src=x onerror="...">`، أول ما الأدمن يفتح الجدول، السكريبت كان بيشتغل **في متصفح الأدمن** ويسرق جلسته ويصدّر كل القاعدة.

**اللي اتعمل:**
- الكود كان فيه بالفعل دالة `escHtml()` مستخدمة في بعض الأماكن.
- وسّعنا التغطية: تم تغليف **77 موضع إضافي** لبيانات المستخدمين المعروضة (الأسماء، الملاحظات، العناوين، اللوحات، الأسباب... إلخ) بـ `escHtml()` في الجداول والكروت ونوافذ العرض — في `app.js` و `driver.js`. إجمالي الاستخدام بقى 100+ موضع.

**اللي لسه ناقص (خطورة أقل):**
- الوسائط داخل `onclick="..."` اللي بتمرّر بيانات مستخدم (نادرة — أغلبها IDs رقمية).
- أي حقول ببيانات مستخدم باسم متغير غير شائع لم يدخل في قائمة التغطية.
- ملاحظة: التغطية الكاملة 100% تحتاج مراجعة يدوية إضافية، لكن أخطر وأشيع مدخل (النص المعروض في الجداول) تم تغطيته.

### 3) توكن UltraMsg مكتوب في الكود
- `app.js`: `ULTRAMSG_INSTANCE="instance178493"` و `ULTRAMSG_TOKEN="eqhb00p3ffkb17zc"`.
- ده سر حقيقي مكشوف — أي حد يبعت واتساب من حسابك ويستهلك رصيدك. **التوكن ده محروق ولازم تعمله rotate فوراً** من لوحة UltraMsg.
- **الحل الصحيح:** نقل نداء UltraMsg لخدمة وسيطة (Cloud Function/سيرفر) بحيث التوكن ما يوصلش المتصفح أبداً.

## 🟠 ثغرات متوسطة

### 4) باسوردات افتراضية / باب خلفي — ✅ تم إصلاحه
- كان فيه حساب مشرف `mutlaq` بباسورد ثابت `123456` بيتعمل تلقائياً (باب خلفي). **تم حذفه.**
- كان فيه باسورد أدمن افتراضي `admin/123`. **تم حذفه.**
- ➜ مطلوب منك: لو كنت بتستخدم حساب `mutlaq`، اعمله من جديد من مركز الإدارة بباسورد قوي.

### 5) تخزين الباسورد نص صريح في المتصفح
- عند "تذكرني" بيتخزن الباسورد في `localStorage` (و`sessionStorage`) نص صريح — قابل للسرقة عبر XSS وبيفضل على الأجهزة المشتركة.
- **الحل:** بعد الانتقال لـ Firebase Auth، نخزّن توكن جلسة مؤقت بدل الباسورد.

### 6) الباسوردات متخزنة نص صريح في القاعدة
- لازم تتشال نهائياً من القاعدة وتتدار عبر Firebase Auth (اللي بيخزّنها مشفّرة بره بياناتك).

## 📋 الأولويات الجاية (أقدر أنفّذها لما توافق)
1. **الانتقال لـ Firebase Authentication** + قواعد أمان + إلغاء تخزين الباسوردات — يقفل الثغرة الحرجة #1 (أهم حاجة متبقية).
2. **نقل UltraMsg (وClaude) لخدمة وسيطة** — يخفي الأسرار فعلياً + عمل rotate للتوكن المحروق.
3. **استكمال XSS** للحالات النادرة المتبقية (وسائط onclick) — خطورة أقل.
4. **ميزة تسجيل المشرفين (approval-gated)** — تتبني بعد #1.

> أهم حاجة متبقية دلوقتي = #1 (Firebase Auth + القواعد). بعدها الداشبورد يبقى آمن فعلياً ضد أغلب محاولات الاختراق.
